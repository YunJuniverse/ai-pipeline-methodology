# 프로젝트 킥오프 프롬프트 템플릿

> 새 프로젝트 시작 시 Claude Code 또는 Codex에 붙여넣기.
> 이 템플릿은 하네스 기반 애자일 방법론에 맞춰, 첫 스프린트를 안전하게 시작하기 위한 입력값을 모은다.

---

## 사용법

1. `./init-project.sh <이름> --type <유형>` 실행
2. 프로젝트 폴더로 이동
3. `CLAUDE.md`와 `AGENTS.md`의 개요 필드를 대략 채움
4. 아래 템플릿을 첫 메시지로 전달

---

## 권장 첫 요청 예시

### 예시 1: fullstack

```text
CLAUDE.md와 AGENTS.md를 읽고 프로젝트를 세팅해줘.

## 아이디어
남성 그루밍 정보를 공유하는 커뮤니티 플랫폼

## 배경
- 왜 지금 이걸 하려고 하는가: 남성 그루밍 시장은 커졌지만 정보와 후기 구조가 파편화되어 있음
- 현재 상태: 아이디어만 있음

## 목표
- 이번 프로젝트의 최종 목표: 커뮤니티 기반 MVP 출시
- 첫 번째 스프린트의 목표: Discovery Research와 Planning Docs v0.1 작성

## 타겟 사용자
- 핵심 사용자: 20~30대 남성, 피부/헤어 관리 정보 탐색 수요가 있는 사용자
- 사용자의 핵심 문제: 남성 중심 정보 허브와 신뢰 가능한 루틴 추천 구조가 부족함

## 레퍼런스
- 화해 — 후기 기반 탐색 구조
- Reddit r/malegrooming — 커뮤니티 운영 흐름
- 오늘의집 — 콘텐츠와 탐색 결합 구조

## 제약 조건
- 예산: 소규모
- 일정: 2개월 내 MVP
- 팀 구성: 1인 개발
- 선호 또는 금지 기술: Next.js + Supabase 선호

## 승인 구조
- 실무 검토자: 본인
- 최종 승인자: 본인
- 승인 받아야 하는 주요 항목: 범위, 주요 기능, 배포 시점

## 오늘 할 일
Discovery Research와 service-overview v0.1, requirements v0.1부터 시작해줘.
```

### 예시 2: coding-only

```text
CLAUDE.md와 AGENTS.md를 읽고 프로젝트를 세팅해줘.

## 아이디어
헤어제품 쇼핑몰 MVP 구현

## 배경
- 왜 지금 이걸 하려고 하는가: 기존 기획이 있고 MVP 구현 단계로 바로 들어가야 함
- 현재 상태: planning docs와 development spec 초안이 있음

## 목표
- 이번 프로젝트의 최종 목표: 결제 전환 가능한 MVP 출시
- 첫 번째 스프린트의 목표: 구현 범위 확정과 MVP v0.1 개발

## 타겟 사용자
- 핵심 사용자: 특정 헤어제품을 빠르게 비교하고 구매하려는 사용자
- 사용자의 핵심 문제: 정보 탐색과 구매 결정이 번거로움

## 레퍼런스
- 올리브영 — 상품 탐색
- 무신사 — 장바구니/주문 흐름

## 제약 조건
- 예산: 제한적
- 일정: 3주
- 팀 구성: 1인 개발
- 선호 또는 금지 기술: Next.js + Supabase + Stripe

## 승인 구조
- 실무 검토자: 본인
- 최종 승인자: 본인
- 승인 받아야 하는 주요 항목: MVP 범위, 결제 정책, 배포 시점

## 오늘 할 일
existing planning docs와 development spec 기준으로 Definition of Ready를 점검하고 Sprint 01 구현 범위를 정리해줘.
```

---

## 입력 가이드

| 항목 | 필요한 이유 |
|------|-------------|
| 아이디어 | 방향을 정하는 최소 입력 |
| 목표 | 최종 목표와 첫 스프린트 목표를 분리하기 위해 |
| 타겟 사용자 | 기획 문서의 기준 사용자 정의 |
| 레퍼런스 | 레퍼런스 괴리 점검 기준선 |
| 제약 조건 | 과도한 설계 방지 |
| 승인 구조 | 인간 승인 하네스를 문서화하기 위해 |
| 오늘 할 일 | 현재 페이즈와 첫 산출물을 결정하기 위해 |

> 최소 입력은 `아이디어`, `레퍼런스`, `오늘 할 일`이다.
> 하지만 이 방법론에서는 `목표`와 `승인 구조`까지 함께 주는 편이 훨씬 좋다.
