# AI 기획 오케스트레이션

## 실행 절차

요청을 받으면 아래 순서로 판단한다.

### 1단계: 요청 해석

**목적 파악:**
- 방향 설정 / 내부 검토 / 대표 보고 / 투자용 / 개발 협업 / 운영 준비 / 런칭 실행

**현재 Stage 파악:**
- CLAUDE.md의 "현재 Stage"와 "현재 상태" 섹션을 먼저 확인한다
- 어떤 문서가 이미 존재하고 어떤 것이 아직 없는지 파악한다

**깊이 파악:**
- 사용자가 명시하지 않으면 **초안** 기본
- "개요만", "구조만", "방향만" → 개요 또는 목차
- "완성본", "실무용", "상세하게", "제대로" → 완성본

### 2단계: 문서 분류

| 요청 키워드 | Stage | 산출물 |
|------------|-------|--------|
| 시장, BM, 수익, 포지셔닝, 고객, 사업성, 투자, 전략 | Stage 1 | foundation.md |
| 브랜드, 톤앤매너, 메시지, 정체성, 비주얼 방향 | Stage 1 | foundation.md (브랜드 섹션) |
| 기능, UX, 시나리오, 정책, IA, MVP, 화면, 권한 | Stage 2 | spec.md |
| 운영, 검수, 제재, CS, 관리자, 프로세스 | Stage 2 | spec.md (운영 섹션) |
| 유입, 전환, 채널, 캠페인, KPI, 리텐션, 광고 | Stage 3 | growth.md |
| 일정, 리스크, 보고, 의사결정, 범위, 마일스톤 | - | CLAUDE.md 현재 상태 섹션 |

### 3단계: 깊이별 가이드라인 소스

**개요~초안:** 이 스킬 내의 압축 가이드라인을 사용한다.

**완성본:** 반드시 원본 지침을 참조한다:
- Foundation → `@docs/planning-guides/10_사업기획서_작성_지침.md` + `@docs/planning-guides/14_브랜드기획서_작성_지침.md`
- Product Spec → `@docs/planning-guides/11_서비스기획서_작성_지침.md` + `@docs/planning-guides/12_운영기획서_작성_지침.md`
- Growth → `@docs/planning-guides/13_마케팅기획서_작성_지침.md`
- 공통 원칙 → `@docs/planning-guides/00_AI_기획_프로젝트_운영_원칙.md`

### 4단계: 복합 요청 처리

하나의 요청이 여러 Stage에 걸칠 경우:
1. **마스터 문서 먼저 정한다** — 요청의 중심 질문이 속한 Stage
2. 보조 문서가 필요한지 판단한다
3. 마스터 → 보조 순서로 작성한다
4. 보조 문서가 지금 안 필요하면 제안만 한다

예시:
- "이 사업을 왜 해야 하는지 + 수익 모델" → Foundation 마스터
- "회원가입부터 후기까지 UX 설계" → Product Spec 마스터
- "운영 기준과 제재 프로세스" → Product Spec (운영 섹션) 마스터
- "플랫폼 전체를 기획해줘" → Foundation 마스터 + Spec 보조 제안

## Foundation (Stage 1) 압축 가이드

### 필수 구조
1. **프로젝트 배경과 목적** — 왜 이 사업을 하는가
2. **시장 환경과 기회** — 시장 규모, 트렌드, 타이밍
3. **타겟 고객 정의** — 핵심 고객 세그먼트, 페르소나
4. **핵심 문제와 솔루션** — 고객의 진짜 문제, 우리의 해결 방식
5. **경쟁 분석과 포지셔닝** — 경쟁자 대비 차별점
6. **비즈니스 모델과 수익 구조** — BM 캔버스, 수익원, 비용 구조
7. **성장 전략과 로드맵** — 단계별 성장 방향
8. **브랜드 방향** [선택] — 정체성, 메시지, 톤앤매너 기본 방향

### 경계
- 기능 상세, 화면 정책 → spec.md로 분리
- 채널별 실행안 → growth.md로 분리
- 운영 절차, 제재 기준 → spec.md 운영 섹션으로 분리

### 브랜드 분리 판단
- 브랜드가 사업의 핵심 차별점이면 Foundation 내 별도 장으로 상세 작성
- 브랜드 방향이 단순하면 Foundation에 1개 섹션으로 충분
- 브랜드 가이드라인(비주얼, 적용 규칙 등)이 필요하면 별도 문서 분리 제안

## Product Spec (Stage 2) 압축 가이드

### 필수 구조
1. **서비스 개요와 목적** — foundation.md의 솔루션을 제품으로 전환
2. **사용자 유형 정의** — 역할별 권한과 행동 범위
3. **핵심 사용자 시나리오** — Happy Path + 예외 + 경계 조건
4. **정보 구조 (IA)** — 메뉴, 화면, 네비게이션 구조
5. **핵심 기능 정의** — Must / Should / Could / Won't Have
6. **정책과 권한** — 서비스 이용 규칙, 역할별 접근 범위
7. **예외처리** — 에러, 엣지케이스, 실패 시나리오
8. **운영 기준** — 검수, 제재, CS 프로세스, 관리자 기능
9. **MVP 범위** — 첫 버전에 포함/제외할 기능
10. **단계별 확장 방향** — MVP 이후 로드맵

### 경계
- 시장성, BM → foundation.md 참조
- 채널 전략 → growth.md로 분리
- 구현 상세 (API, 데이터 모델) → Stage 4에서 처리

### 운영 섹션 분리 판단
- 운영 규모가 크면 (CS, 검수, 제재가 핵심) spec.md 내 별도 파트로 상세화
- 운영이 단순하면 기능 정의에 운영 규칙을 인라인 처리
- 운영 매뉴얼이 필요한 수준이면 별도 문서 분리 제안

## Growth (Stage 3) 압축 가이드

### 필수 구조
1. **성장 목표와 KPI** — 측정 가능한 목표 수치
2. **타겟 사용자 세분화** — foundation.md의 고객을 마케팅 타겟으로 세분화
3. **유입 채널 전략** — 채널별 역할, 우선순위, 예산 배분
4. **전환 구조** — 퍼널 단계별 전환율 목표
5. **리텐션 전략** — 재방문, 재구매, 커뮤니티 활성화
6. **캠페인 설계** — 런칭 캠페인, 시즌 캠페인
7. **측정 체계** — 트래킹 툴, 대시보드, 리포팅 주기

### 경계
- 브랜드 메시지 원본 → foundation.md 참조
- 기능 상세 → spec.md 참조
- 운영 프로세스 → spec.md 운영 섹션 참조

## 공통 품질 체크리스트

작성 완료 후 아래를 확인한다:

- [ ] 목적이 첫 부분에 1문장으로 드러나는가
- [ ] 독자(누가 읽는가)가 명시되어 있는가
- [ ] 범위(어디까지)가 분명한가
- [ ] 추상어 없이 구체적인가
- [ ] 다른 문서와 경계가 보이는가
- [ ] 실행 또는 판단에 도움이 되는가
- [ ] 중복이 과하지 않은가
- [ ] 재사용 가능한 구조인가
