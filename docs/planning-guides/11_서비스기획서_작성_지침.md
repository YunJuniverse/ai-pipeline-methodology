# 11_서비스기획서_작성_지침

## 1. 문서 개요

### 1.1 문서명
서비스기획서 작성 지침

### 1.2 문서 목적
본 문서는 AI가 서비스기획서를 작성할 때 따라야 하는 표준 작성 기준을 정의한다.  
서비스기획서는 사용자 경험, 기능 구조, 정보 구조, 정책, 권한, 예외처리를 실제 구현 가능한 수준으로 정리하는 문서다.

### 1.3 문서의 역할
서비스기획서는 사업기획서의 상위 방향을 실제 사용자 경험과 제품 구조로 전환하는 문서다.  
즉, 이 문서는 사용자가 어떤 흐름으로 서비스를 이용하는지, 어떤 기능이 필요한지, 어떤 정책과 예외 기준 아래 서비스가 동작하는지를 정의한다.

### 1.4 적용 범위
본 지침은 아래 유형의 문서 작성에 적용한다.

- 서비스기획서

- PRD

- IA 문서

- 사용자 시나리오 정의 문서
- 정책 중심 서비스 설계 문서

## 2. 서비스기획서의 정의

### 2.1 기본 정의
서비스기획서는 **사용자 경험과 서비스 기능 구조를 실제 구현 가능한 수준으로 정의하는 문서**다.  
이 문서는 서비스가 어떤 목적을 가지고, 누가 어떤 흐름으로 사용하며, 어떤 규칙과 정책 아래 동작하는지를 정리한다.

### 2.2 이 문서가 다루는 핵심
서비스기획서는 아래를 다룬다.

- 사용자 유형
- 사용자 시나리오
- 서비스 구조
- 정보 구조
- 핵심 기능
- 정책
- 권한
- 예외처리
- MVP 범위
- 단계별 확장 방향

### 2.3 다른 문서와의 차이
서비스기획서는 사업기획서처럼 시장성, BM, 재무를 원본으로 다루지 않는다.  
운영기획서처럼 검수 절차, 제재 단계, CS 처리 프로세스를 상세 원본으로 다루지 않는다.  
마케팅기획서처럼 채널, 퍼널, 캠페인 전략을 중심으로 다루지 않는다.  
브랜드기획서처럼 메시지 체계와 톤앤매너를 원본으로 관리하지 않는다.

서비스기획서는 **사용자가 실제로 접하는 구조와 기능과 정책의 원본 문서**다.

### 2.4 한 문장 정의
서비스기획서는 **“사용자가 무엇을 어떤 흐름으로 어떻게 이용하게 될지를 정의하는 구현 기준 문서”**다.

## 3. 서비스기획서의 목적

### 3.1 핵심 목적
서비스기획서의 목적은 아래와 같다.

1. 서비스의 전체 구조를 사용자 관점에서 정의한다
2. 필요한 기능과 정책을 정리한다
3. 디자이너와 개발자가 구현 가능한 수준의 기준을 제공한다
4. 운영기획과 QA, 후속 협업의 기준이 되는 문서를 만든다

### 3.2 이 문서가 해결하는 문제
서비스기획서가 없으면 다음 문제가 발생하기 쉽다.

- 기능이 많지만 사용자 흐름이 정리되지 않음
- 화면은 있는데 정책이 없음
- 정책은 있는데 권한과 예외 기준이 없음
- 개발팀마다 해석이 달라짐
- 운영 기준과 기능 구조가 충돌함
- MVP와 후속 기능의 구분이 안 됨

### 3.3 필요한 시점
아래 단계에서 서비스기획서가 필요하다.

- 사업 방향이 정리된 후 제품 구조를 설계할 때
- MVP 범위를 확정할 때
- 디자인 착수 전 기능 구조를 정리할 때
- 개발 착수 전 정책과 예외를 정리할 때
- 운영기획과 QA 기준의 상위 입력값을 만들 때
- 기존 서비스 리뉴얼 구조를 재정리할 때

## 4. 핵심 질문

AI는 서비스기획서를 작성할 때 반드시 아래 질문에 답해야 한다.

### 4.1 사용자는 누구인가
- 어떤 유형의 사용자가 존재하는가
- 사용자별 목적과 기대는 무엇인가
- 권한에 따라 어떤 차이가 있는가

### 4.2 사용자는 어떤 흐름으로 서비스를 이용하는가
- 처음 진입부터 핵심 액션까지 어떤 흐름을 밟는가
- 주요 use case는 무엇인가
- 반복 사용 흐름은 어떻게 설계되는가

### 4.3 어떤 기능이 필요한가
- 핵심 문제 해결에 필요한 기능은 무엇인가
- 기능 간 관계는 어떻게 되는가
- 어떤 기능이 필수이고 어떤 기능이 후순위인가

### 4.4 어떤 정책과 권한이 필요한가
- 회원/비회원/관리자/파트너 등은 무엇을 할 수 있는가
- 등록, 수정, 삭제, 공개, 비공개 조건은 무엇인가
- 승인, 제한, 신고 관련 기준은 무엇인가

### 4.5 어떤 예외처리가 필요한가
- 실패 상황은 무엇인가
- 제한 상황은 무엇인가
- 사용자가 막히는 케이스는 무엇인가
- 시스템이 어떻게 반응해야 하는가

### 4.6 MVP 범위는 어디까지인가
- 반드시 포함해야 하는 핵심 기능은 무엇인가
- 나중으로 미룰 수 있는 기능은 무엇인가
- 단계별 확장 구조는 어떻게 되는가

## 5. 주 독자별 기대 포인트

### 5.1 PM / PO
- 상위 사업 방향이 실제 기능 구조로 정리되었는가
- MVP 범위가 명확한가
- 정책과 권한이 빠지지 않았는가
- 운영과 개발로 자연스럽게 연결되는가

### 5.2 디자이너
- 사용자 흐름이 분명한가
- 어떤 화면이 필요한가
- 각 화면의 목적은 무엇인가
- 권한과 정책에 따른 UX 분기가 보이는가

### 5.3 개발자
- 기능 정의가 명확한가
- 입력값/출력값/조건이 보이는가
- 예외처리가 정리되어 있는가
- 권한과 상태값이 분명한가

### 5.4 QA
- 정상 흐름과 예외 흐름이 모두 정의되어 있는가
- 테스트 포인트가 보이는가
- 정책 기준이 검증 가능한 형태인가

### 5.5 운영 협업 담당자
- 운영과 충돌하는 정책이 없는가
- 관리자 개입 지점이 보이는가
- 후속 운영기획서 작성의 기반이 되는가

## 9. 항목별 작성 기준

### 9.1 서비스 개요 작성 기준
반드시 포함:
- 서비스 목적
- 한 줄 정의
- 해결하려는 문제
- 이번 문서 범위

권장 질문:
- 이 서비스는 사용자에게 어떤 가치를 주는가
- 이 문서에서 어디까지 정의할 것인가

### 9.2 사용자 정의 작성 기준
반드시 포함:
- 사용자 유형
- 사용자 목적
- 역할
- 권한 차이

권장 방식:
- 단순 “회원/비회원”을 넘어서 실제 행동 기준으로 구분한다
- 사용자마다 기대 액션을 함께 정리한다

### 9.3 사용자 시나리오 작성 기준
반드시 포함:
- 시작 조건
- 주요 행동 흐름
- 목표 액션
- 종료 조건

권장 방식:
- 시나리오는 기능보다 행동 흐름 중심으로 쓴다
- 핵심 시나리오와 보조 시나리오를 구분한다

### 9.4 IA 작성 기준
반드시 포함:
- 상위 메뉴
- 하위 메뉴
- 콘텐츠 구조
- 탐색 관계

권장 방식:
- 서비스 전체를 한눈에 볼 수 있게 작성한다
- 사용자가 어디서 어디로 이동하는지 연결성을 드러낸다

### 9.5 기능 정의 작성 기준
반드시 포함:
- 기능명
- 기능 목적
- 사용 주체
- 핵심 액션
- 결과

권장 방식:
- 기능을 나열하지 말고 목적과 연결한다
- 유사 기능은 군집화한다

### 9.6 기능 상세 명세 작성 기준
반드시 포함:
- 진입 조건
- 동작 방식
- 입력 조건
- 처리 로직
- 결과 상태
- 제한 조건

권장 방식:
- 정상 흐름과 예외 흐름을 함께 본다
- 가능한 경우 상태 변화 관점을 포함한다

### 9.7 화면 정책 작성 기준
반드시 포함:
- 화면 목적
- 주요 정보 요소
- 핵심 액션
- 진입/이탈 조건

권장 방식:
- 와이어프레임 설명서가 아니라, 화면의 역할과 정책을 중심으로 쓴다

### 9.8 정책 정의 작성 기준
반드시 포함:
- 등록 가능 조건
- 수정 가능 조건
- 삭제 가능 조건
- 공개 범위
- 제한 조건
- 권한 차이

권장 방식:
- 정책은 반드시 조건문 수준으로 구체화한다
- “관리자 판단”으로 뭉개지지 않게 쓴다

### 9.9 예외 처리 작성 기준
반드시 포함:
- 어떤 상황이 예외인지
- 사용자에게 무엇을 보여줄지
- 시스템은 어떻게 반응할지
- 후속 조치가 필요한지

권장 방식:
- 예외는 나중에 쓰는 부록이 아니라 핵심 설계 요소로 본다
- 실패 상황과 권한 이슈를 빠뜨리지 않는다

### 9.10 권한 체계 작성 기준
반드시 포함:
- 역할 구분
- 역할별 가능한 액션
- 제한 액션
- 관리자 개입 필요 지점

권장 방식:
- 권한표 형태로 작성하면 좋다
- 사용자 유형과 권한을 혼동하지 않는다

### 9.11 MVP 범위 작성 기준
반드시 포함:
- MVP 필수 기능
- 후순위 기능
- 제외 기능
- 향후 검토 기능

권장 방식:
- MVP는 “최소 기능 나열”이 아니라, 핵심 가치 검증 범위로 정의한다
- 왜 제외하는지 이유도 붙이면 좋다

## 10. 작성 원칙

### 10.1 사용자 흐름을 중심에 둔다
서비스기획서는 기능 나열 문서가 아니다.  
사용자가 어떤 맥락에서 어떤 흐름으로 서비스를 이용하는지를 중심으로 작성해야 한다.

### 10.2 기능보다 목적을 먼저 쓴다
각 기능은 반드시 왜 필요한지가 먼저 설명되어야 한다.

### 10.3 정상 흐름과 예외 흐름을 함께 본다
정상 시나리오만 적으면 개발과 QA와 운영에서 충돌이 생긴다.  
예외 상황, 제한 상황, 실패 상황을 반드시 포함해야 한다.

### 10.4 정책은 모호하지 않게 쓴다
“필요 시 제한한다” 같은 표현보다, 어떤 조건에서 무엇을 제한하는지 구체적으로 적는다.

### 10.5 권한과 상태를 분리해서 본다
권한은 누가 할 수 있는가의 문제고, 상태는 현재 객체나 프로세스가 어떤 상황인가의 문제다.  
둘을 혼동하지 않게 작성해야 한다.

### 10.6 MVP와 확장을 명확히 나눈다
모든 아이디어를 한 번에 넣지 않는다.  
핵심 가치 검증에 필요한 범위를 우선 분리한다.

### 10.7 다른 문서와 연결될 수 있어야 한다
서비스기획서는 운영기획, 마케팅기획, 디자인, 개발, QA로 이어질 수 있는 중간 허브 문서여야 한다.

## 11. 자주 하는 실수

### 11.1 기능만 나열하고 시나리오가 없는 실수
기능 목록은 있지만 사용자가 어떻게 흐르는지 없으면 UX 구조가 무너진다.

### 11.2 화면 설명만 있고 정책이 없는 실수
화면은 있는데 등록, 수정, 삭제, 공개, 권한 기준이 없으면 구현과 운영이 흔들린다.

### 11.3 정책은 있는데 예외가 없는 실수
정상 상황만 있으면 실제 서비스에서 빈틈이 커진다.

### 11.4 사업기획 내용을 반복하는 실수
시장성, BM, 경쟁 분석을 길게 반복하면 문서 목적이 흐려진다.

### 11.5 운영기획과 경계가 없는 실수
서비스 정책과 운영 절차를 구분하지 않으면 문서가 과도하게 복잡해진다.

### 11.6 MVP 기준이 없는 실수
기능은 많은데 우선순위가 없으면 실행이 늦어진다.

### 11.7 권한 체계가 빠지는 실수
권한 정의가 없으면 개발과 운영에서 해석이 달라진다.

## 12. 다른 문서와의 경계

### 12.1 사업기획서와의 경계
서비스기획서는 사업 방향을 입력값으로 받지만, 시장 분석과 수익 구조를 원본으로 다루지 않는다.

### 12.2 운영기획서와의 경계
서비스기획서는 서비스 내 정책과 권한을 정의할 수 있으나, 운영팀의 검수 절차와 제재 프로세스 원본은 운영기획서가 담당한다.

### 12.3 마케팅기획서와의 경계
서비스기획서는 랜딩, 가입, 전환 관련 서비스 포인트를 정의할 수 있으나, 채널 전략과 캠페인 실행안은 마케팅기획서가 담당한다.

### 12.4 브랜드기획서와의 경계
서비스기획서는 브랜드 경험 반영 포인트를 일부 포함할 수 있으나, 메시지 체계와 표현 원칙의 원본은 브랜드기획서가 담당한다.

### 12.5 프로젝트 관리 기획서와의 경계
서비스기획서는 기능 구조와 정책의 원본을 정의하고, 프로젝트 관리 기획서는 해당 서비스기획 산출물의 일정, 검토, 승인, 변경과 같은 실행 관리 구조를 담당한다.

## 13. 좋은 문장 패턴

### 13.1 서비스 정의 패턴
본 서비스는 [핵심 사용자]가 [핵심 문제]를 해결할 수 있도록 [핵심 기능 구조]를 제공하는 서비스다.

### 13.2 사용자 시나리오 패턴
사용자는 [시작 상황]에서 서비스에 진입한 뒤, [주요 행동]을 수행하고, 최종적으로 [목표 결과]에 도달한다.

### 13.3 기능 목적 패턴
본 기능의 목적은 [문제/목적]을 해결하기 위한 것으로, [사용자]가 [행동]을 수행할 수 있도록 지원한다.

### 13.4 정책 정의 패턴
[사용자 유형]은 [조건]을 만족할 경우에만 [행동]이 가능하며, [예외 조건]에 해당하는 경우 해당 액션은 제한된다.

### 13.5 예외 처리 패턴
사용자가 [예외 상황]에 해당할 경우 시스템은 [반응]을 수행하며, 사용자에게는 [안내 방식]으로 안내한다.

### 13.6 MVP 패턴
MVP 단계에서는 [핵심 가치 검증]에 필요한 기능만 우선 포함하고, [후순위 기능]은 후속 단계에서 검토한다.

## 14. 금지 패턴

### 14.1 추상적인 UX 설명
예:
- 편리한 사용자 경험을 제공한다
- 직관적인 화면 구성을 지향한다
- 자연스러운 플로우를 설계한다

이런 문장은 구체 구조 없이 단독 사용 금지

### 14.2 화면 이름 나열
예:
- 홈
- 마이페이지
- 설정
- 리뷰 등록

이런 식의 목록만으로는 문서 가치가 낮다  
각 화면의 목적과 정책이 함께 있어야 한다

### 14.3 정책 없는 기능 설명
예:
- 리뷰를 등록할 수 있다
- 게시글을 수정할 수 있다

이런 문장은 누가, 언제, 어떤 조건에서 가능한지 빠져 있으므로 불완전하다

### 14.4 예외 없는 정상 흐름만 설명
실제 서비스 문서에서는 실패 케이스가 반드시 필요하다

### 14.5 운영 절차 과다 삽입
예:
- 신고 접수 후 관리자 A가 1차 검수하고 24시간 내 처리
이런 내용은 운영기획서 원본으로 분리

## 16. 품질 체크리스트

작성 후 아래를 점검한다.

- 서비스 목적이 첫 부분에 명확한가
- 사용자 유형이 충분히 구체적인가
- 핵심 시나리오가 기능보다 먼저 보이는가
- IA와 기능 구조가 연결되는가
- 기능마다 목적이 드러나는가
- 정책과 권한이 명확한가
- 예외 처리와 실패 상황이 포함되어 있는가
- MVP와 후속 기능이 구분되는가
- 운영기획과 서비스기획의 경계가 보이는가
- 디자인/개발/QA가 실제로 활용 가능한 수준인가

## 17. AI 실행용 축약 지시문

앞으로 서비스기획서를 작성할 때, 이를 사용자가 무엇을 어떤 흐름으로 이용하게 될지를 정의하는 구현 기준 문서로 취급하라.  
먼저 사용자 유형, 핵심 시나리오, 서비스 구조, IA, 주요 기능, 정책, 권한, 예외처리, MVP 범위를 정의하라.  
기능은 나열하지 말고 사용자 목적과 흐름 중심으로 설명하라.  
정상 흐름뿐 아니라 실패 상황과 제한 조건도 반드시 포함하라.  
상세 시장 분석과 재무 구조는 사업기획서로, 운영 절차 원본은 운영기획서로, 채널 실행안은 마케팅기획서로 분리하라.  
문서는 디자인, 개발, QA, 운영 협업이 가능한 수준으로 구체화하라.
