## 1. 파이프라인 원칙

### 문서 = 다음 단계의 프롬프트
모든 문서는 산출물인 동시에 다음 단계 AI 작업의 입력이다.
Foundation → Product Spec → Growth → Implementation 순으로, 이전 문서가 다음 문서의 컨텍스트가 된다.

### 경계 유지
동일 정보의 원본은 한 문서에만 존재한다.
다른 문서에 필요하면 요약 또는 참조로 연결한다.

### 구조 중심 작성
문장보다 구조가 먼저. 목차, 표, 분류 체계, 우선순위를 먼저 세운다.

### 추상어 금지
"혁신적인", "차별화된", "사용자 친화적인" 등 단독 사용 금지.
반드시 근거 또는 구체 항목을 붙인다.

### 실행 가능성 우선
멋진 표현보다 판단 기준, 실행 절차, 책임 주체, 우선순위, KPI가 보이는 문서.

### 멱등성
같은 요구사항 → 항상 같은 구조로 수렴.
파일/함수 중복 생성 방지. DB는 upsert. 환경변수 처리. 에러 핸들링 포함.

### Human-in-the-loop
AI 단독으로 아키텍처/전략 결정 금지. 반드시 승인 후 진행.

## 2. 파이프라인 Stage

### Stage 0: Project Init
- CLAUDE.md 프로젝트 개요 작성
- 필요 Stage 선택 (프로젝트 유형별 프리셋 참고)

### Stage 1: Foundation (WHY)
- 시장, 고객, 문제, 포지셔닝, BM, 브랜드 방향
- 산출물: `docs/foundation.md`
- 매핑: 사업기획서 + 브랜드기획서(방향)

### Stage 2: Product Spec (WHAT)
- 사용자 흐름, 기능, 정책, 운영 기준, MVP 범위
- 산출물: `docs/spec.md`
- 매핑: 서비스기획서 + 운영기획서 + PRD

### Stage 3: Growth (HOW TO GROW) [선택]
- 유입, 전환, 리텐션, 채널, KPI
- 산출물: `docs/growth.md`
- 매핑: 마케팅기획서

### Stage 4: Implementation (HOW TO BUILD) [코딩 프로젝트만]
- 4-레이어 아키텍처, TDG 루프, 린터/커밋훅
- spec.md가 PRD 역할을 겸함 — 별도 CPS/PRD 작성 불필요
- 구현 순서: domain → application → infrastructure → interface

### 프로젝트 유형별 Stage 프리셋

| 유형 | Stage | 설명 |
|------|-------|------|
| fullstack | 0→1→2→3→4 | 기획부터 코딩까지 |
| planning-only | 0→1→2→3 | 기획 산출물만 |
| research | 0→1 | 리서치/사업성 검토만 |
| coding-only | 0→2→4 | 기존 기획 기반 바로 코딩 |

## 3. 깊이 제어 (Depth Control)

모든 문서는 5단계 깊이로 작성 가능하다.

| 깊이 | 설명 | 가이드라인 소스 |
|------|------|----------------|
| 개요 | 1페이지 요약 | 스킬 내 압축 가이드라인 |
| 목차 | 구조 + 섹션별 한 줄 설명 | 스킬 내 압축 가이드라인 |
| 초안 | 주요 섹션 채워진 문서 | 스킬 내 압축 가이드라인 |
| 완성본 | 실무 사용 가능 수준 | @docs/planning-guides/ 원본 참조 |
| 템플릿 | 재사용 가능한 빈 구조 | @docs/planning-guides/ 원본 참조 |

기본값: **초안**. 사용자가 명시하지 않으면 초안 수준으로 작성.

## 5. 코드 컨벤션 (Stage 4 전용)

> 코딩 프로젝트가 아니면 이 섹션을 무시한다.

### 네이밍
- 파일명: 복수 명사 (`users.ts`, `products.ts`)
- 금지 접미사: `List`, `Detail`, `Manager`, `Helper`
- 컴포넌트/클래스: PascalCase | 함수/변수: camelCase
- 상수/환경변수: UPPER_SNAKE_CASE
- 타입: PascalCase, `I` 접두사 금지

### 구조 규칙
- 함수 50줄 초과 시 즉시 분리
- 중첩 3단계 초과 금지 → early return
- `any` 타입 절대 금지 | `unknown` + 타입 가드
- `console.log` 프로덕션 코드 잔류 금지
- Export: barrel file(`index.ts`) 사용

### 커밋 컨벤션
- `feat`: 새 기능 | `fix`: 버그 수정 | `refactor`: 동작 변경 없는 개선
- `docs`: 문서 수정 | `test`: 테스트 | `chore`: 빌드/설정

### 금지 사항
- `any` 타입 | 전역 상태 무분별 사용 | 하드코딩 시크릿
- 레이어 경계 위반 import | 테스트 없는 비즈니스 로직 함수
